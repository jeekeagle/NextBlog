## Agent越来越灵活，人越来越需要约束

以 Hermes 为代表的 Agent 产品，设计方向是**充分自由**——自然语言交互、工具调用、记忆持久化、能力可扩展。理论上，你可以让 Agent 做任何事。

但人的使用习惯恰恰相反。

人在面对一个强大工具时，倾向于两种极端：**要么不用，要么滥用**。不用是因为不知道从哪里开始；滥用是因为能力太强而边界太模糊。两者的根源是同一个——没有在人与工具之间建立一套**共同的语境体系**。

这个语境体系，包括三个需要主动构建的维度：

- **话语体系**：哪些词触发什么动作，哪些说法代表什么意图
- **文件体系**：信息存在哪里，以什么结构组织，流动方向是什么
- **思考认知体系**：收到什么信息，走什么流程，给出什么输出

这三个维度，不是让 Agent 去适应人的习惯，而是**人与 Agent 共同建构一套新的协作语言**。

---

## 核心分析：充分自由是协作的第一障碍

一个设计良好的 Agent，应该能理解自然语言。但自然语言的歧义性、多义性，使得「充分自由」在实践中变成「充分混乱」。

举一个实际的例子：用户说「整理一下」。

对人来说，这句话可能意味着：整理文件、清理笔记、回顾项目、输出报告。对 Agent 来说，没有上下文就完全无法判断。Agent 会按自己的理解执行——可能和用户预期的完全不同。

解决这个问题，不是让 Agent 更聪明（LLM 已经足够聪明），而是**让人和 Agent 在「整理」这个词上达成共识**。

这就是 router.md 存在的意义——把「整理」这个模糊词汇，翻译成具体的路由规则。用户说「整理笔记」，Agent 知道走 Get笔记处理流程；用户说「整理文章」，Agent 知道走 Clippings 处理流程。

**不是 Agent 理解不了，是人没有说清楚。**

Meta 配置层的工作，就是把人说清楚。

## Meta 配置的三个核心组件

### Zone 模式：划定专注范围

Hermes 是一个全能 Agent，但人不是全能的。人写作时想的是发布流程，实验时想的是技术验证，碎片记录时想的是捕捉灵感。三种场景需要三种不同的注意力焦点。

Zone 模式通过模式切换，把 Hermes 的「当前上下文」固定到特定场景。用户说「切换到 OpenWalk 模式」，Hermes 加载的是文章发布的语境——发布流程、GitHub 同步规则、作者署名规范。这避免了「写篇文章」时 Agent 突然开始帮你分析健康数据。

本质上，Zone 模式是在**给 Agent 划定一个行动边界**。边界之内，Agent 有充分的自由；边界之外，Agent 不会擅自行动。

### 信息路由器：翻译模糊意图

router.md 是 Meta 层的核心。它不是代码，但比代码更重要。

它解决的是这样一个问题：用户说一句自然语言，Agent 需要在多个可能的处理路径中做出选择。选择错误，用户的体验就是「Agent 听不懂我说话」。

18 条路由规则，把用户可能说的话和对应的处理流程一一映射。当用户说「发一条 post」，Agent 不是去猜测意图，而是查 router.md，找到规则 18，走 Get笔记保存流程。

**这不是限制 Agent 的能力，是在给 Agent 一张地图。** 没有地图的 Agent 到处跑，有了地图的 Agent 知道往哪走。

### 工具矩阵：防止接完就忘

matrix.md 记录了 Hermes 已接入哪些外部服务、什么时间接入、有哪些定时任务。

它的作用是**提供全局视图**。一个外部工具接入后如果没记录，过一段时间人自己都不记得用过这个工具。matrix.md 任何时候都能回答：Hermes 现在能做什么外部操作。

---

## 文件体系的协同价值

Obsidian 下的 N 系列目录结构，本质上是 Hermes 的「外部记忆」。

Agent 的上下文窗口是有限的，但文件体系是无限的。把信息外存到 N 系列目录，Hermes 就可以处理超出上下文容量的复杂信息——它从文件里读，不需要全存在脑子里。

更重要的是，**文件体系是人与 Agent 共同维护的**。人在 Obsidian 里写笔记，Hermes 在 Wiki 里建知识条目；人发布文章到 OpenWalk，Hermes 同步到 GitHub Pages。这个共同维护的过程，是「人-Agent 协作语言」最直接的外化。

四大 Wiki（ai/health/next/ocean）的铁律——不可删除、不可移动，只可修改——也是一种协作约定：**知识的完整性优先于灵活性**。这条约束对 Agent 比对人更重要，因为 Agent 更容易在「优化」的名义下删除它认为无用的信息。

---

## 适用边界：Meta层不是什么

Meta 层解决的是「同一件事重复做」的问题。如果一件事只做一次，为它写路由规则反而是浪费。Meta 层适用于**高频、重复、有明确模式**的信息处理任务。

Meta 层也不解决能力边界问题。它可以规定「遇到代码问题走 software-development 技能」，但不能提升 Agent 写代码的能力。能力问题要靠模型升级和技能扩展，Meta 层解决的是**组织问题**——让已有的能力在正确的位置被调用。

---

## 结论

人与 Agent 的协作范式，不应该是「人发指令，Agent 执行」的线性关系。

真正有效的协作，是**人构建秩序，Agent 在秩序内自由发挥**。

Meta 配置层，就是这套秩序的具体载体。它不是限制 Agent，而是让 Agent 的能力真正可以被依赖、可被预测、可持续运转。

当你发现「Hermes 好像听不懂我说什么」的时候，问题通常不在 Agent，在 Meta 层——缺少一条路由规则，缺少一个 Zone 模式，缺少一个文件结构的约定。

Meta 层是人与 Agent 协作的**第一性建设**。把它搭好，Agent 的能力才会真正释放出来。
