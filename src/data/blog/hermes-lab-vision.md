我们用 GitHub 工具的成本，一直都很高。

看到一个有意思的项目，克隆，读代码，搭环境，部署，测试，踩坑，判断。走完这一趟，一整天没了。大部分时候，踩坑踩到一半，项目就被判定为「没价值」，扔掉。

这不是方法问题，是经济模型问题。

## 判断力被浪费在哪里

我们不缺判断力。在这个行业待久了，一个工具有没有价值、能不能提升效率、有没有长期维护的潜力——是有感觉的。但这种感觉用不上，因为被一个更前置的问题消耗了：**这个工具能不能跑起来都不确定，我怎么判断它的价值？**

判断力是认知行为的燃料。但燃料是有限的。它被用在「部署测试」上，就没空用在「价值判断」上。搭环境踩的每一个坑，都是认知资源的消耗。而大多数时候，真正缺的恰恰是后者。

不是找不到好工具。是找到了，也用不上。

## 执行者困境

有一个概念叫**执行者困境**：人在解决问题时，花太多时间做执行工作（搭环境、对依赖、解决奇怪的报错），直到某个时刻才"灵光乍现"进入真正的判断状态。

GitHub 工具消费就是这个困境的典型场景。

人本来应该做判断者：看到新工具，说「这个方向有意思，值得研究」。但现实是，人被迫先做执行者：先让工具跑起来，再说别的。而执行这件事，恰好是 AI 的强项。

**把执行外包给 AI，把判断留给自己。** 这是 Hermes Lab 存在的底层逻辑。

## 分离原则

GitHub 工具的消费，可以拆成三个阶段：**探索期**（这个工具是什么）、**验证期**（它能不能用）、**整合期**（值不值得集成进工作流）。

人在三个阶段的能力分布是不均匀的。探索期需要快速判断方向，整合期需要战略眼光——这两项是人的强项。验证期需要的是耐心和执行力——这两项恰好是 AI 的强项，而且 AI 不会因为反复踩坑而产生挫败感。

但我们一直把验证期当成主战场，花最多时间。对一个工具的判断，往往在「能不能跑起来」这个阶段就被决定了，而不是在「值不值得用」这个阶段。

**判断和验证是两件不同的事。** 判断是认知行为——这个东西有没有价值，符不符合需求，长期风险有多高。验证是执行行为——这个工具能不能跑起来，输出是不是正确。混在一起做的代价是：验证失败时，判断也跟着死掉了。

分离之后，人把精力用在判断上，AI 把精力用在验证上。这不是分工，这是认知资源的重新配置。

## Hermes Lab 的职责

Hermes Lab 是 Hermes 的探索实验室，它的职责是把「这个工具值不值得用」的验证成本降下来。

用户说「帮我测一下 Quarkdown」，Hermes 就去做：拉取源码、分析架构、判断平台兼容性、验证能否构建、输出结论。用户基于结论做判断——用还是不用，怎么用。

**人提出判断目标，AI 负责验证过程。** 这是一个分工，不是一个替代。

这和 DRY 原则是同一件事。DRY 原则说「不要重复自己」，把重复性的记忆和操作交给外部系统；Hermes Lab 说「不要重复消耗判断力」，把执行层的验证交给 AI。区别只在于：DRY 外包的是操作，Hermes Lab 外包的是认知。

## 增量入口

Hermes 有很多模块——知识库、写作系统、发布流程、定时任务。这些都是存量，已经确定要用的能力，运行稳定，不需要经常动。

Hermes Lab 是增量入口。

这可以用 **15%原则** 来理解：保持 85% 的能力稳定，用 15% 持续探索新工具。每测试一个工具，Hermes 的能力边界就往外扩一点。上一次测试积累的经验——踩过的坑、发现的边界条件、平台兼容性的判断框架——下次测试新工具时可以直接复用。Hermes 不是在重复劳动，它在**积累**。

存量的模块决定 Hermes 现在能做什么。增量的入口决定 Hermes 未来能做什么。一个系统能走多远，取决于它的增量入口够不够宽。

## 四种结局

一个工具进入 Hermes Lab，最终只有四种结局：

**集成** — 工具被 skill 化，成为 Hermes 能力的一部分。这是目标结局，但不是唯一正确的结局。

**归档** — 工具不适合，但记录了为什么不适合，供以后参考。这也是有效输出。知道一个工具「为什么不行」，和知道它「为什么行」一样有价值。

**丢弃** — 工具没有价值，快速结束，不消耗更多资源。快速失败是一种能力，不是失败。

**等待** — 工具现在不成熟，但方向有价值，记下来，等生态成熟再测。等待也是一种判断。

没有「一直测但没有结论」这个选项。每一次探索，都要有输出。

## 边界

Hermes Lab 有它解决不了的问题。

平台强依赖的工具，需要特定显卡、特定硬件外设——Hermes 的执行环境是 Linux 服务器，模拟不了物理机需求。

纯图形界面的工具，没有 CLI，只有图形操作——这类工具不是 Hermes 的盲区，是 Hermes 的禁区。

需要账号授权的工具，Hermes 能验证部署可行性，但无法验证业务可行性。

Hermes Lab 擅长的是：**有源码、有 CLI、能在服务器环境跑起来的工具**。这是它的边界，也是它的特色。

## 高视角：判断力的稀缺性

Hermes Lab 表面上在解决「如何高效测试工具」的问题。

它真正在解决的，是**认知资源的配置问题**。

判断不是猜，而是基于经验和洞察做出的高质量选择。好的判断需要充沛的认知资源作支撑。当认知资源被消耗在部署环境、依赖冲突、平台兼容性这些执行层的事情上，判断的质量就会下降——人会变得保守，开始回避需要深入评估的工具，或者干脆跳过判断直接放弃。

这才是 GitHub 工具消费困境的本质：不是工具不够多，是人的认知资源被错误的消耗结构浪费了。

人的认知资源是有限的。97% 的担忧是被夸大的，有限的认知资源应该省着用，用在真正需要判断的地方。Hermes Lab 把验证从判断中剥离出来，本质上是把「执行层」外包给 AI——这是 DRY 原则在认知层面的应用：让机器做重复性工作，解放头脑去专注真正需要思考的事情。

一个使用 AI 辅助工作的人，判断力才是真正的瓶颈。

工具在变多。认知资源是稀缺的。两者之间的接口，才是最重要的设计。
