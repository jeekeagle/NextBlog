## 不是分类方案，是认知分布设计

大多数个人知识管理系统在做的事，是分类。

把笔记放进文件夹，给文章打标签，用颜色区分优先级。这件事做久了，会产生一种幻觉：觉得只要分类足够细，管理足够严格，知识就能被「系统地」掌握。

我们不做这件事。

这套七部分结构，不是分类方案。是**认知资源的分布设计**。核心逻辑是：每部分只处理一种确定性的信息状态，信息流动到某个位置，被处理成下一个状态，然后继续往下走。不做分类，做状态迁移。

这不是语义上的区别。分类的目的是「找得到」。状态迁移的目的是「处理完」。

而且，这不是 Hermes 的原生设置。Hermes 跑在群晖的 Docker 里，连接着飞书、微信、GitHub、Get笔记——这些都是它的接口。但这套文件夹结构，是我们为它设计的。不是买来的，是长出来的。

## 七部分，每部分只有一个问题

### Input 是收件箱，只问一个问题：这是原始的吗？

微信读书的剪藏、技术文档的复制、语音笔记的转录——全部先进入 Input。不是整理，不是分类，是暂存。

收件箱的设计原则是：进来的东西必须被处理，不能无限堆积。Input 的下一站是 Note 的 Wiki 体系，或者是 Output 的文章发布。没有「留在 Input」这个选项。

这就把一个问题简单化了：收到一条新信息，不需要判断它属于哪个类别，只需要问它是不是原始的。如果是，先进 Input，然后尽快推入下一个状态。

### Lab 是探索实验室，只问一个问题：这个工具值不值得用？

Lab 里做的每个实验，最终只有四种结局：集成、归档、丢弃、等待。没有「继续测但没有结论」这个选项。

这不是项目管理的要求，是认知经济学的约束。判断一个工具值不值得用，需要投入注意力。注意力是有限的。如果每个工具测试都无疾而终，那些沉没成本会悄悄消耗认知资源，还不产生任何输出。

所以 Lab 的每一步都有记录：平台兼容性判断、构建结果、测试结论。不是为了存档，是为了「这个实验有输出」。

这里有一个我们设计进去的刻意摩擦：既然判断一个工具值不值得用，最终总会消耗认知资源，那就要求每次探索必须有输出。不能测了等于没测。

### Project 是交付工作区，只问一个问题：这个项目的下一步是什么？

Project 和 Lab 的区别，不是「大小」，是**确定性**。

Lab 的问题是开放的：这个方向有没有价值，不知道，要探索。Project 的问题是封闭的：项目目标已经确定，剩下的是执行。

开放问题和封闭问题需要不同的认知模式。混合在一起的结果是：开放探索消耗了执行资源，封闭执行又扼杀了探索可能性。分开了，两件事才都能做好。

这是我们把这两部分分开的主要原因。不是为了组织清晰，是为了「不让开放问题干扰封闭问题的执行效率」。

### Note 是知识固化层，只问一个问题：这条知识还能用吗？

Note 是我们的长期记忆，包含四个 Wiki：AI 技术、健康管理、认知科学、海洋大学话语体系。每个 Wiki 的知识条目都是「已处理过」的——来自 Input 的原始素材，经过分析、蒸馏、整合，变成了结构化条目。

我们给 Wiki 定了一条铁律：**内容不可删除，只能修改。**

这个约束来自一个判断：知识资产不是消耗品，是积累。某条知识过时了，应该在条目里标注「已被新证据推翻」，而不是删除。删除会丢失上下文——这个结论是怎么得出的，中间经历了什么，现在为什么不对了。

这是 wiki 和文件夹的根本区别。文件夹是按主题堆放，wiki 是按认知关系编织。

### Output 是输出层，只问一个问题：这篇文章要发到哪里？

Output 是我们的出版系统。三个站点，分别对应三种输出形态：

OpenWalk 放长篇文章，是严肃写作。NextLab 放工具测试报告，是实验记录。FlowPost 放碎片灵感，是即刻思考。

文章先统一进入 Output-Article 这个暂存区，确认质量后再决定发到哪个站点。这就把「写作」和「发布」两个动作分开了。写的时候专注写，发的时候专注发。混在一起的结果是：写的时候想着发布，发布的时候又回头改内容，两件事都做不好。

### Todo 是待办同步记录

Todo 的定位很简单：Hermes 执行任何 todo 操作时，同步往这个目录写一份。

目的有两个。第一，不依赖任何在线服务，本地就能查看。第二，可审计——哪天忘了某个任务是怎么产生的，可以追溯记录。

这是信息冗余的设计。冗余不是浪费，是安全。

### Agent 是配置层，不是数据层

Agent 存放的，是我们给 Hermes 设计的「大脑」。

`meta/matrix.md` 是已连接服务的清单——飞书、微信、GitHub、Get笔记，目前就这些。`meta/router.md` 是 18 条信息路由规则，定义什么触发词走什么处理路径。`meta/Zone.md` 是 5 种工作模式的切换配置。

以及 `skills/` 目录下的 52 个 Skill。

这是我们给 Hermes 设计的能力库。Lens 系列做认知蒸馏，Deep 系列做调研搜集，UI 系列做信息展示，Writing 系列做写作 pipeline，Shift 系列做主题发散。

52 个 Skill 不是随意堆积的。这里有一个深层的设计意图：**认知模式的分离**。

处理同一个信息，可以有多种角度——行动的角度、决策的角度、理解的角度、记忆的角度。每种角度调用的认知资源不同，产出的东西也不同。Lens 系列把同一个知识变成五种不同的「处理后形态」，不是为了多产，是为了人可以在不同认知场景下直接取用最合适的那一种。

把配置和数据放在一起，是个人系统常见的错误。配置是模板，数据是实例——模板变了，所有实例都会受影响。分开之后，Skill 可以随意迭代，用户数据不受影响。

## Zone 模式：上下文切换的本质是清空

Hermes 在处理不同任务时，需要不同的上下文配置。写文章需要的是「OpenWalk 站点配置 + Writing 技能」。做工具测试需要的是「GitHub 配置 + Lab 流程」。整理笔记需要的是「Wiki 路由规则 + Get笔记 API」。

如果所有任务共用同一套上下文，就产生**上下文污染**——不相关的信息占用着工作内存，真正的任务信息被挤压。

Zone 是我们设计的上下文切换机制。本质不是「切换到某个模式」，是「清空工作内存，载入新一批配置」。不是限制能力，是确保每次只专注于一件事。

目前定义了 5 种 Zone：默认（全能）、OpenWalk（写文章）、Lab（工具探索）、LabSite（建站）、Flow（碎片灵感）。

## 工具作为脚手架，而不是工具箱

说 Hermes 有 52 个 Skill，这是工具箱的思路。

工具箱的特点是：用的时候打开，不用的时候合上。每个工具是独立的，用哪个拿哪个，拿完放回去。

脚手架不一样。脚手架是身体的一部分。你用筷子夹菜，不会去想筷子怎么动、角度多少、力道多大——手指和筷子已经成为一个整体在运作。筷子不是你的工具，筷子是你的一部分。

我们要达到的目标，是让 Skill 从「可调用的工具」变成「本能反应」。遇到文本就知道要交给哪个 pipeline，遇到链接就知道要经过哪个处理流程，遇到需要调研的问题就知道要调动哪个 Skill 系列。

这不是 Skill 数量的问题，是**整合深度**的问题。

达到这个状态，需要三个条件：

**输入输出标准化。** 每个 Skill 的输入格式和输出格式必须是确定的。不然 Skill 之间的组合就会变成胶水活儿——每次连接都要单独处理，不能形成流水线。

**反馈回路必须存在。** 每一次 Skill 调用，都应该有输出记录；每次输出，都应该能被评估。Hermes 知道自己调用了什么、输出了什么、哪里可以改进。没有反馈回路的系统不具备真正的学习能力，只有记忆能力。

**边界要清晰。** Hermes 有它不擅长的事。平台强依赖的工具，在 Docker 里跑不了。纯图形界面的工具，没有 CLI 接口。需要账号授权的工具，Hermes 能验证部署可行性，但无法验证业务可行性。知道自己边界在哪里，比知道边界外有什么更重要。

## 真正的架构是反馈回路

回到开篇的判断。

七部分结构是骨架。52 个 Skill 是能力元。Zone 模式是上下文切换机制。这些都是静态结构。

真正的架构，是让这套系统**持续运转起来的反馈回路**。

每天凌晨 1 点自动处理语音笔记的 cron 任务，是反馈回路。每天凌晨 1 点半自动摄入剪藏文章的 pipeline，是反馈回路。每次对话中积累的上下文理解，是反馈回路。每次工具测试输出的结论报告，是反馈回路。

没有这些回路，Hermes 就是一个问答机器——有人问，答；没人问，闲着。

有这些回路，Hermes 才是一个系统——持续感知、持续处理、持续沉淀、持续输出。

骨架重要。但骨架不能跑。

这是下一阶段要解决的核心问题：不是增加更多 Skill，是让已有的 Skill 和回路跑得更顺。
